01 / PRODUCT & BEHAVIOR

How it thinks about your code.

What does Pliny actually do on a pull request?

The moment a PR opens, Pliny clones the diff into an ephemeral runner, replays your test commands against the changed surface, then reads the diff with the rest of your codebase as context. It posts review comments inline — concrete suggestions with patches you can click-apply — and a top-level summary that flags risk areas, missing tests, and regressions against recently-merged behavior.

It does not approve PRs on its own. It comments like a careful senior would: pointed, occasionally wrong, never sycophantic.

Is this just GPT-4 with a prompt that says “review this code”?

No. The frontier model is one of seven components. The pipeline includes a static call-graph indexer (we index your repo on install and incrementally on every push), a regression classifier trained on ~3.8M reverted commits, a test-impact analyzer that runs only the suites a diff can plausibly break, and an opinion layer that's tuned per-team based on which suggestions you accept or dismiss.

The LLM writes the prose. Everything that decides what to talk about is deterministic.

How is this different from CodeRabbit, Greptile, or GitHub Copilot reviews?

Three things, honestly. First, we don't comment on style — your linter does that better and we'd rather not be noise. Second, we run your tests; most competitors only read the diff. Third, Pliny learns per-repo: dismiss a class of suggestion three times and it stops making it. Most tools treat every PR like the first PR they've ever seen.

What languages and frameworks are supported?

First-class support: TypeScript, JavaScript, Python, Go, Rust, Ruby, Java, Kotlin, Swift, C#, PHP, Elixir. Frameworks with bespoke heuristics: React, Next.js, Vue, Django, Rails, Phoenix, FastAPI, Spring Boot, Laravel, ASP.NET.

C and C++ are in private beta — DM us if you want in. Terraform and Kubernetes manifests are reviewed but with narrower test impact.

Will Pliny hallucinate suggestions that reference functions that don't exist?

Rarely, and never silently. Every suggestion is grounded in an AST node the indexer can resolve back to a file and line. If the model proposes calling resolveUserSession(), we check that the symbol exists, is in scope, and has a compatible signature before posting. If any of those fail, the comment is dropped — we'd rather skip a thought than ship a hallucination.

Our internal hallucination rate, measured on a held-out set of 12k PRs, is currently 0.31%.

02 / SECURITY & DATA

Where your code goes, exactly.

Do you train on our code?

No. Not for the base model, not for any shared model, not ever. The indexes and per-repo opinion tuning live in an isolated tenant and are deleted on disconnect within 30 days (or immediately, on request).

If you self-host, your code never leaves your VPC. We can't train on it because we can't see it.

Is Pliny SOC 2 compliant?

SOC 2 Type II audit is in progress with Prescient Assurance, completion expected Q2 2025. Our Type I report is available under NDA — security@pliny.dev. ISO 27001 is on the roadmap for late 2025.

Can we self-host?

Yes. The self-hosted edition ships as a Helm chart that runs on any Kubernetes cluster >= 1.27. You bring the inference — OpenAI, Anthropic, Bedrock, Azure, or your own vLLM endpoint via the BYOM adapter. The indexer, runner, and orchestrator stay inside your cluster.

Median install time across our design-partner cohort is 41 minutes.

What about secrets in our codebase?

Detected and redacted before the diff is ever sent to a model — AWS keys, GCP service accounts, private keys, JWTs, and 140+ other patterns from the trufflehog rule set, plus a learned classifier for shapes we haven't seen before. Secrets that slip through are flagged as a P0 review comment, never echoed back into the suggestion text.

03 / INTEGRATION

Fitting into how you already work.

What git providers do you support?

GitHub Cloud and GitHub Enterprise Server (3.9+) today. GitLab (SaaS and self-managed) and Bitbucket Cloud are in private beta — expected GA in May. Gitea and Forgejo are on the roadmap, no committed date.

How do I tell Pliny our team's conventions?

Drop a .pliny/conventions.md in your repo root. Plain English, no DSL. “We don't use try/except for control flow.” “Prefer pure functions in /lib.” “Database migrations go through Atlas, never raw SQL.” Pliny re-reads it on every push and weights it heavily in the opinion layer.

You can also pin specific files as “canonical examples” — Pliny will model new code after them.

Can I tell it to shut up about something?

Yes — three ways. React 👎 to a comment in the PR and it learns. Add a topic to .pliny/ignore (e.g. “naming conventions”, “import ordering”). Or @-mention @pliny mute thread to silence a specific conversation. All three persist across PRs.

04 / PRICING & ACCESS

Beta, billing, and what comes next.

What does it cost?

During public beta: free for teams under 12 engineers, $24/engineer/month above that, capped at $480/repo/month. Self-hosted is a flat $1,800/month per cluster, BYOM (you pay the inference provider directly).

Public pricing locks for 12 months from your install date — we won't surprise you.

I'm in. How long until I'm actually using it?

From requesting access to first PR comment: typically 36 hours. We're reviewing the waitlist by hand right now (sorry) to make sure we onboard teams we can actually support. Initial indexing of an average 400k-LOC repo takes 8–14 minutes.
