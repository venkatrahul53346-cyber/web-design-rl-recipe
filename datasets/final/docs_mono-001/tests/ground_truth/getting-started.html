Getting started

From openapi.yaml to a typed client in under five minutes.

This walkthrough installs the CLI, generates a TypeScript client from a real spec, boots the conformant mock server, and runs the producer side of a contract test. Roughly 90 seconds of typing, plus whatever your package manager decides to do.

read time4 min node≥ 18.17 cliv0.14.2

Install the CLI

Glyph ships as a single static binary with no runtime dependencies. The recommended install uses the official script — it picks the right archive for your platform and verifies the checksum against the release manifest before unpacking into ~/.glyph/bin.

macOS / linux windows (powershell) homebrew npm
install.shshell
$ curl -fsSL https://glyph.dev/install.sh | sh
# resolving release manifest for darwin-arm64 ........ ok
# verifying sha256 (a31cf2…) .......................... ok
# installing to ~/.glyph/bin/glyph .................... ok

$ glyph --version
glyph 0.14.2 (darwin-arm64, built 2024-11-04)

Prefer a package manager? Glyph is published to Homebrew (brew install glyph-dev/tap/glyph) and npm (npm i -g @glyph/cli). The npm package wraps the same binary, so version pinning works across both channels.

Air-gapped environment? Download the release tarball from GitHub, drop the binary anywhere on $PATH, and skip the script. Glyph never phones home outside of glyph update.

Your first spec

If you already have an OpenAPI 3.x document, point Glyph at it. Otherwise, scaffold a minimal spec to follow along — glyph init writes a workspace config and a tiny petstore.yaml next to it.

terminalshell
$ mkdir petstore && cd petstore
$ glyph init --template=minimal
  created  glyph.toml
  created  openapi/petstore.yaml
  created  .gitignore

$ glyph check
  ✓ openapi/petstore.yaml — 3.1.0, 2 paths, 3 schemas
  ✓ no breaking changes vs. baseline (none recorded)
  ✓ ready

glyph check is the single command worth memorizing. It validates the spec, lints style rules, and — if a baseline exists — diffs the current version against the last committed contract. The exit code is non-zero on any failure, which is why CI loves it.

Generate a typed client

Targets are configured in glyph.toml. The init template registers one TypeScript client out of the box; here we add a Rust client beside it. Every target writes to a deterministic path so generated code stays diff-friendly across runs.

glyph.tomltoml
[spec]
source  = "openapi/petstore.yaml"
baseline = ".glyph/baseline.lock"

[[target]]
kind     = "client-ts"
out      = "clients/ts"
runtime  = "fetch"     # or "axios", "undici"

[[target]]
kind     = "client-rs"
out      = "clients/rs"
runtime  = "reqwest"

Then run the generator. Glyph is idempotent — re-running with no spec change is a no-op and a no-write.

$ glyph generate
  ✓ clients/ts  — 12 files, 8 operations, 0 unchanged
  ✓ clients/rs  — 9 files,  8 operations, 0 unchanged
  done in 412ms

The TypeScript client is plain ESM, zero runtime dependencies beyond fetch, and every operation is typed end-to-end including discriminated error responses. Here's the hello-world consumer:

examples/hello.tstypescript
import { createClient } from "../clients/ts";

const api = createClient({ baseUrl: "http://localhost:4010" });

const result = await api.pets.list({ limit: 10 });

if (result.ok) {
  // result.data is typed as Pet[], not any.
  console.log(result.data.map(p => p.name));
} else {
  // result.error is the discriminated error union.
  console.error(result.error.code, result.error.message);
}

Run the mock server

The same spec drives a conformant mock. Glyph's mock honours examples, x-glyph-stateful hints, and response selection via the Prefer header — so consumer tests can exercise the unhappy paths without producer changes.

$ glyph mock --port 4010
  serving openapi/petstore.yaml on http://localhost:4010
  ▸ GET    /pets         → 200 (example: "two-pets")
  ▸ POST   /pets         → 201
  ▸ GET    /pets/{id}    → 200, 404
  ready · ctrl-c to stop

Hit it with the generated client above and you have a working consumer against an authoritative fake — no producer required, no hand-rolled fixtures to drift.

A contract test on the producer side

On the producer, point Glyph at the running service and ask it to replay every operation in the spec, including negative cases. The exit code is the contract.

$ glyph verify --target http://localhost:8080
  running 14 cases against openapi/petstore.yaml
  ✓ GET    /pets                    200      18ms
  ✓ GET    /pets?limit=10           200      12ms
  ✓ POST   /pets {valid}            201      24ms
  ✗ POST   /pets {missing name}     500 ≠ 400  9ms
  ✓ GET    /pets/42                 200      11ms
  ✓ GET    /pets/does-not-exist     404       8ms

  1 failed · contract breach on POST /pets
  exit 2
Wire this into CI early. Run glyph check on every PR and glyph verify in your service's integration job. The pair catches the vast majority of "we shipped a breaking change and nobody noticed" incidents at review time.
