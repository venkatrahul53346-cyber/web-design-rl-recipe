Example 01

Typed TypeScript client from spec

use-case · consumer output · fetch + zod runtime · node 20, bun, deno

What this does

Generates a fully-typed client from openapi.yaml with request validation, response parsing, and discriminated error unions. Schemas land in ./src/gen/; nothing is written outside that directory.

The --runtime fetch flag produces zero-dependency code. Pass --validator zod when you want runtime checks at the boundary.

Tip. Commit the generated output. Glyph's diff command compares it to the next regeneration and refuses to overwrite untracked edits.
src/billing.tstypescript
// generated facade re-exports the typed client
import { createClient, ApiError } from "./gen/billing";

const billing = createClient({
  baseUrl: "https://api.acme.io/v3",
  headers: { authorization: `Bearer ${process.env.ACME_KEY}` },
  validator: "zod",        // parse responses, throw on shape mismatch
});

export async function chargeCustomer(id: string, cents: number) {
  const res = await billing.invoices.create({
    body: { customer_id: id, amount_cents: cents, currency: "USD" },
  });

  if (res.ok) return res.data;       // Invoice — fully typed

  switch (res.error.kind) {
    case "insufficient_funds":
      return { retry: true, after: res.error.retry_after_ms };
    case "customer_archived":
      throw new ApiError("archived", res.error);
    default:
      throw new ApiError("unexpected", res.error);
  }
}
$ glyph generate client \
    --spec ./openapi.yaml \
    --lang typescript --runtime fetch --validator zod \
    --out ./src/gen/billing
→ 47 operations · 132 schemas · wrote 18 files · 0 unsafe casts
Example 02

Stand up a mock server in Go tests

use-case · consumer tests output · httptest handler runtime · go 1.22+

What this does

Spawns a hermetic mock backend from the spec for the duration of a test. Responses are sampled from examples in the OpenAPI file; missing examples fall back to schema-driven fakers seeded for determinism.

The mock validates outgoing requests against the spec, so if your code sends a malformed payload the test fails with a precise pointer like #/components/schemas/Invoice/properties/currency.

billing_test.gogo
package billing_test

import (
    "net/http/httptest"
    "testing"

    "github.com/glyph-dev/glyph-go/mock"
    "acme.io/billing"
)

func TestChargeCustomer_RetriesOnInsufficientFunds(t *testing.T) {
    srv := httptest.NewServer(mock.FromSpec(t, "openapi.yaml",
        mock.Override("POST /invoices", mock.Response{
            Status: 402,
            Body:   map[string]any{"kind": "insufficient_funds", "retry_after_ms": 1500},
        }),
        mock.ValidateRequests(true),
    ))
    defer srv.Close()

    client := billing.New(srv.URL)
    got, err := client.Charge("cus_42", 2500)

    if err != nil {
        t.Fatalf("unexpected: %v", err)
    }
    if !got.Retry || got.AfterMs != 1500 {
        t.Errorf("expected retry=true after=1500, got %+v", got)
    }
}
Why not a recorded fixture? Fixtures rot the moment the spec changes. The mock here is rebuilt from the spec on every test run — if the field renames, the test fails at the assertion, not three weeks later in staging.
Example 03

Fail CI on breaking contract diffs

use-case · producer guardrails output · github actions check runtime · any

What this does

On every pull request, Glyph compares the head spec against main and classifies the diff. Removing a field, narrowing a type, or tightening a required-list is breaking and exits non-zero. Adding an optional field is additive and passes.

The annotations show up inline on the PR — the engineer sees exactly which line in openapi.yaml broke which downstream consumer.

.github/workflows/contract.ymlyaml
name: contract
on: { pull_request: { paths: ["openapi.yaml"] } }

jobs:
  diff:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }

      - uses: glyph-dev/setup-glyph@v1
        with: { version: "0.14.2" }

      - name: diff against main
        run: |
          glyph diff \
            --base origin/main:openapi.yaml \
            --head openapi.yaml \
            --policy strict \
            --report github-annotations \
            --consumers ./glyph.consumers.toml

      - name: post summary
        if: always()
        run: glyph diff --format markdown >> $GITHUB_STEP_SUMMARY
$ glyph diff --base main --head HEAD --policy strict
BREAKING  openapi.yaml:412  required field removed
  paths./invoices.post.requestBody.schema.required
  − "currency"
  affects consumers: web-checkout, mobile-ios, partner-stripe
exit 1 · 1 breaking · 2 additive
Example 04

Detect schema drift against production

use-case · observability output · drift report runtime · python 3.11+

What this does

Samples real responses from a deployed environment for a configurable window, validates each one against the published spec, and emits a drift report grouped by endpoint and field. Useful when you suspect a service is returning fields that aren't documented — or missing fields that are.

Validation is streaming; you can point it at a Kafka topic of captured responses with --source kafka://... instead of replaying traffic.

tools/drift_audit.pypython
from glyph import Spec, Sampler, DriftReport

spec = Spec.load("openapi.yaml")

sampler = Sampler(
    target="https://api.acme.io",
    auth=("bearer", os.environ["OPS_TOKEN"]),
    sample_rate=0.02,         # 2% of live responses
    duration="15m",
    redact=["$.customer.email", "$..ssn"],
)

report = DriftReport()
async for hit in sampler.stream():
    result = spec.validate_response(
        path=hit.path, method=hit.method,
        status=hit.status, body=hit.body,
    )
    if not result.ok:
        report.add(hit, result.errors)

report.write("drift.html")
print(f"drift: {report.endpoints_affected} endpoints, {report.total_errors} mismatches")
Common finding. Optional fields the spec marks as string but the service occasionally returns as null. Glyph flags these as nullable drift and proposes the one-line spec patch in the report.
Example 05

Talk to the Glyph daemon over JSON-RPC

use-case · editor / tooling output · RPC client runtime · rust 1.74+

What this does

The Glyph daemon (glyphd) holds parsed specs in memory and exposes incremental operations over a local JSON-RPC socket. Editors and pre-commit hooks use this instead of invoking the CLI per-call — a watched 4k-operation spec validates in under 8ms.

Below: a Rust snippet that asks the daemon which operations in openapi.yaml changed semantically since the last commit. Useful for selective test runs.

crates/affected/src/main.rsrust
use glyphd_client::{Client, DiffSince};
use std::path::PathBuf;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let client = Client::connect("unix:///run/user/1000/glyphd.sock").await?;

    let spec = client
        .load_spec(PathBuf::from("openapi.yaml"))
        .await?;

    let diff = client
        .diff_since(DiffSince {
            spec_id: spec.id,
            git_ref: "HEAD~1".into(),
            include_additive: false,
        })
        .await?;

    for op in diff.affected_operations {
        println!("{:8} {} {}", op.severity, op.method, op.path);
    }
    Ok(())
}
